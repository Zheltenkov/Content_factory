## D1 — Граница входного уровня (`assumed_known`)

**Контекст.** Без явной границы спиральный планировщик либо раздувает УП до абсолютной базы, либо
вешает дыры, требующие сторонних источников (что 3.2.2 запрещает).

**Решение.** `assumed_known` — явное, ссылается на каталог (52 скилла), по правилу из регламента 3.1.1:
- **Вход/бассейн:** `assumed_known = []` (новичок с нуля). Опц. минимальный baseline «общая ИТ-грамотность» — методолог подтверждает per-направление.
- **«Ветка из ветки»:** `assumed_known = union(компетенции, формируемые предыдущими ветками)` — тянется из УП-DAG/каталога.

Скилл `audience_level` (planner-стадия) **produces** в контекст:
```yaml
assumed_known: [competency_id, ...]     # пусто для бассейна; outcomes предыдущих веток иначе
experienced_share: 0.30                  # эвристика 3.1.1, НЕ порог — калибрует сложность
level_source: pool | branch_of_branch
```
Планировщик начинает граф от фронтира `assumed_known`; компетенции на/ниже границы не переобучаются.

**Обоснование.** Привязка к той же компетентностной сети делает границу проверяемой и data-driven;
закрывает обе крайности (переобучение базы / дыры). Прямо следует из 3.1.1.

**[подтвердить]** Есть ли единый baseline «entry literacy» для бассейна, или всегда `[]`.

## D2 — Гранулярность графа

**Решение.** Граф на уровне **компетенций (скиллов)** на старте; концептный подслой **отложен**.

**Обоснование.** Регламент, каталог (52 скилла), XP-распределение и веса — все говорят в единицах
компетенций. Концептный слой сейчас спекулятивен и не подкреплён данными. Добавляется позже как
атрибутный слой **аддитивно**, без реструктуризации. Рёбра — prerequisite на уровне компетенций.

**Что пинит.** `core/models` graph-узел = competency; `branch_structure`/`competency_weights` работают
на этой грануляции. Никакого concept-узла в схеме сейчас.

## D3 — Калибровка часов

**Решение.** LLM оценивает **одну величину — человеко-часы X**. Всё остальное — детерминированно в коде
(принцип «модель оценивает одно, формулы считают остальное», как в nutrition-app):
```
calendar_days = round(X * 0.34) + 3      # 0.34 — индекс (5 дн/нед × 4 ч/день); +3 дня на проверку
xp            = X * 10                     # из листа УП [подтвердить против живого листа]
```
- X — единственная оценка, дискретна, в полосе листа `[hours_band]` **[подтвердить: 8–24?]**.
- Дедлайны — отдельно: проекты группируются в **11 периодов** по сумме трудоёмкости + факторы (3.1.3).
- Контроль качества: **3 проверки на проект** (мин. 2) — «золотое правило».

Скилл `workload_planning` (planner): `check.py` валидирует X в полосе, пересчитывает days/xp, флагует
рассинхрон. Params:
```yaml
days_index: 0.34
review_days: 3
xp_per_hour: 10
hours_band: [8, 24]        # [подтвердить]
reviews_per_project: 3     # min 2
```

**Обоснование.** Минимизирует степени свободы модели до одной оценимой величины; days/xp — чистые
функции, не «оцениваются» и не дрейфуют. Формулы из 3.1.3 (`Y=X·0.34+3`) и листа УП согласованы
(0.34 ≈ 1/2.94).

## D4 — Состав жюри + слаги OpenRouter

**Решение (правило).** Дидактическое жюри (PoLL):
- **≥3 гетерогенных модели** (разные вендоры/семейства) — гетерогенность и есть то, что снижает
  single-model bias; нечётное число для устойчивости медианы.
- `GENERATOR_MODEL` **исключён** из жюри (анти-self-bias).
- `DEBATE_ROLES` (critic/defender/judge) — разные модели, отличные друг от друга и от генератора.

**Рекомендованный дефолт** (вендор-разнообразно, до точной калибровки): жюри = {модель Claude} +
{модель GPT} + {модель Gemini}, при генераторе на Qwen/DeepSeek-стеке (тогда жюри их не включает).
Состав финально подбирается **по измеренному согласию с экспертом** (QWK/Spearman/ICC) — это
калибровочный долг (SKILLS_ARCHITECTURE §10), не угадывание.

**Что пинит.** Конфиг checker (волна 5):
```yaml
GENERATOR_MODEL: <slug>                 # [подтвердить — какой слаг генератора]
JURY_MODELS: [<slug1>, <slug2>, <slug3>] # ≥3, вендор-разные, ≠ GENERATOR_MODEL  [подтвердить слаги]
DEBATE_ROLES: {critic: <slug>, defender: <slug>, judge: <slug>}  # разные  [подтвердить]
```

**[подтвердить]** Доступные тебе слаги OpenRouter + слаг `GENERATOR_MODEL`. Это единственное решение,
где нужен твой факт; правило и дефолт-семейства зафиксированы.

## D5 — Справочник: общий PG vs за контрактом

**Решение.** Справочник — в **общем Postgres с самого старта** (его схема и так переезжает в общую
alembic-цепочку на T0.4). Контракт `ProfilePackage`/`UPSkeleton` обязателен в любом случае, но как
**in-process типизированная граница** (сервисный интерфейс), а не сетевой API над отдельным хранилищем.

**Обоснование.** Смысл консолидации — закрыть CSV-шов, теряющий идентичность навыка. Временный API над
отдельным стором воссоздал бы шов (просто типизированный). Раз данные и так в общем PG, «за контрактом
над отдельным стором» — лишняя работа против консолидации. Контракт остаётся как типовой интерфейс
каталог↔генератор внутри процесса.

**[подтвердить]** Используется ли `Spravochnik/viewer` в живом проде, который нельзя поставить на паузу
на время миграции. Если да — временный read-through на переходный период; если это твой инструмент —
сразу общий PG.
